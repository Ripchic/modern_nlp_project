# Design Doc: ShopAdvisor — LLM-based помощник по выбору товаров

> Основан на шаблоне команды Reliable ML
> Заполняется проектной командой: бизнес-заказчик, ML/инженер, продуктовая команда

---

## 1. Контекст проекта

### 1.1 Бизнес-задача

**Проблема:** При выборе товара (гаджет, бытовая техника, снаряжение и т.д.) пользователь
вынужден самостоятельно агрегировать информацию из десятков источников: YouTube-обзоры,
форумы (Reddit, Otzovik, 4PDA), статьи-сравнения, сайты магазинов. Это занимает часы
времени и требует экспертизы для фильтрации шума.

**Решение:** Telegram-бот (или Mini App), которому пользователь скидывает ссылки и/или
произвольный текст — бот парсит контент, строит базу знаний по теме и выдаёт
структурированную рекомендацию: плюсы / минусы / итоговый вывод "что купить".

**Почему это имеет смысл:**
- Экономия времени: 2-4 часа ресёрча → 2-3 минуты
- Снижение когнитивной нагрузки: не надо быть экспертом
- Персонализация: рекомендация под конкретный запрос пользователя

**Критерии успеха (бизнес):**
- Пользователь получает ответ быстрее, чем самостоятельный ресёрч
- Рекомендация совпадает с "экспертным мнением" в ≥70% случаев (ручная оценка)
- NPS ≥ 7/10 на тестовой группе из 10 человек

---

### 1.2 Целевая аудитория и пользователи

**Кто:** Частные пользователи, выбирающие товары для личного использования.
Студенты, молодые специалисты (18-35 лет), активные пользователи Telegram.

**Сценарии использования:**
1. Пользователь пишет в бот: "Хочу купить наушники до 5000 руб"
2. Скидывает 2-5 ссылок (YouTube, статья, форум)
3. Бот парсит → строит контекст → выдаёт: плюсы/минусы каждого варианта + итоговая рекомендация
4. Пользователь задаёт уточняющие вопросы: "А если мне нужно для спорта?"

**Интерфейс:** Telegram Bot (основной) + опционально Telegram Mini App для красивой карточки.

**Нагрузка:**
- MVP: 1-5 активных пользователей (команда + друзья)
- Целевая: до 50 сессий/день
- Пиковые часы: вечер, 19:00-23:00

---

### 1.3 Ограничения и допущения

**Не делаем в рамках проекта:**
- Не интегрируемся с реальными магазинами (нет парсинга цен в реальном времени)
- Не обрабатываем PDF и изображения (только текстовый контент веб-страниц и YouTube-субтитры)
- Не храним историю пользователей между сессиями (MVP)
- Не обрабатываем закрытый контент (paywall, авторизация)

**Допущения:**
- Пользователь скидывает 2-10 ссылок за сессию
- YouTube-видео имеют субтитры (автоматические или ручные)
- Контент на русском или английском языке
- Домашний CPU-сервер доступен 24/7, стабильное интернет-соединение

**Репутационные риски:**
- Бот может дать неверную рекомендацию → предупреждение в каждом ответе:
  "Это не экспертный совет, финальное решение за вами"
- На вопросы не по теме бот отвечает: "Я помогаю только с выбором товаров"

## 2. Архитектура решения

### 2.1 Общая схема

```
[Пользователь в Telegram]
        │  POST: ссылки + вопрос
        ▼
[Telegram Bot (python-telegram-bot)]
        │  HTTP POST /api/v1/analyze
        ▼
[REST API — FastAPI]  ←── единая точка входа
        │
        ├──▶ [Content Fetcher]
        │         ├── YouTube Transcript API
        │         ├── trafilatura (статьи, форумы)
        │         └── plain text
        │
        ├──▶ [Chunker + Embedder]
        │         └── sentence-transformers (CPU)
        │
        ├──▶ [MongoDB]  ←── хранилище сессий, чанков, истории, логов
        │         ├── sessions        (сессии пользователей)
        │         ├── chunks          (векторы + текст чанков)
        │         ├── interactions    (логи запросов/ответов)
        │         └── feedback        (👍/👎 от пользователей)
        │
        ├──▶ [Retriever]  ←── MongoDB Vector Search (Atlas) или cosine по chunks
        │
        ├──▶ [Prompt Builder]
        │
        ├──▶ [LLM — GPT-4o mini]  ←── GitHub Models API (основной)
        │         └── Gemini 2.0 Flash  ←── резерв (Google AI Studio, бесплатно)
        │
        └──▶ [Post-processor + Formatter]
                  │  JSON response
                  ▼
        [Telegram Bot] → отправляет ответ пользователю
```

**REST API эндпоинты:**

| Метод | Эндпоинт | Описание |
|---|---|---|
| `POST` | `/api/v1/analyze` | Принять ссылки + вопрос, запустить RAG pipeline |
| `POST` | `/api/v1/sessions` | Создать новую сессию |
| `GET` | `/api/v1/sessions/{id}` | Получить историю сессии |
| `DELETE` | `/api/v1/sessions/{id}` | Удалить сессию и её чанки |
| `POST` | `/api/v1/feedback` | Сохранить фидбэк (👍/👎) |
| `GET` | `/api/v1/health` | Health check |

**Порядок обработки запроса:**
1. Telegram-бот получает сообщение → POST `/api/v1/analyze`
2. REST API создаёт/находит сессию в MongoDB (`sessions`)
3. Content Fetcher парсит каждую ссылку → чистый текст
4. Chunker нарезает текст на чанки → Embedder векторизует
5. Чанки с векторами сохраняются в MongoDB (`chunks`)
6. Retriever ищет топ-5 релевантных чанков по вектору в��проса
7. Prompt Builder собирает промпт: system + context + user query
8. Запрос к LLM API (GitHub Models / Gemini fallback)
9. Ответ форматируется → сохраняется в `interactions`
10. JSON ответ → Telegram-бот → пользователь

---

### 2.2 Хранилище знаний / документооборот

**Источники знаний:**
- YouTube: субтитры через `youtube-transcript-api`
- Веб-страницы: статьи, форумы, обзоры — через `trafilatura`
- Произвольный текст от пользователя

**Предобработка:**
- Удаление HTML-тегов, рекламных блоков, навигации (trafilatura)
- Нормализация пробелов и спецсимволов
- Фильтрация коротких чанков (< 50 токенов → drop)
- Удаление дубликатов (cosine similarity > 0.95 → drop)

**Индексация:**
- Embedding модель: `sentence-transformers/paraphrase-multilingual-mpnet-base-v2`
  (мультиязычная RU+EN, работает на CPU, ~420 MB)
- Размер чанка: 512 токенов, overlap: 50 токенов
- Метрика: cosine similarity
- Метаданные чанка: `{source_url, source_type, chunk_index, session_id, created_at}`

**Обновление данных:**
- База — сессионная: новая сессия = новые документы в коллекции `chunks`
- Сессии TTL 24 часа (MongoDB TTL index) → автоматическая очистка
- Если пользователь добавляет ссылки в рамках сессии — инкрементально добавляем

### 2.3 Интеграции и интерфейсы

**Внешние сервисы:**
- `Telegram Bot API` — интерфейс с пользователем
- `GitHub Models API` — GPT-4o mini (основной LLM, OpenAI-совместимый REST)
- `Google AI Studio API` — Gemini 2.0 Flash (резервный LLM)
- `youtube-transcript-api` — субтитры
- `trafilatura` — парсинг веб-страниц

**Протоколы:**
- Telegram: Webhook (HTTPS через Nginx) в prod / Long Polling в dev
- GitHub Models / Gemini: REST API (JSON)
- Внутри: FastAPI REST между ботом и backend

**UI/UX ответа в Telegram:**
```
📦 Анализ завершён по 3 источникам

✅ Плюсы Sony WH-1000XM5:
• Лучшее шумоподавление в классе
• До 30 часов автономной работы

❌ Минусы:
• Цена выше среднего
• Пластиковый корпус

🏆 Рекомендация: Sony — лучший выбор, если бюджет позволяет.
Если нет — рассмотри Anker Q45.

⚠️ Это не экспертный совет. Финальное решение — за вами.
```
Inline-кнопки: `[🔍 Уточнить]` `[👍 Полезно]` `[👎 Неточно]`

**Мониторинг:**
- Все запросы/ответы логируются в MongoDB (`interactions`)
- Ошибки парсинга логируются в `interactions.errors`
- При критических ошибках → Telegram-уведомление разработчику

### 2.4 Инфраструктура и развёртывание

**Стек:**
- **Домашний CPU-сервер** — основная платформа
- **Docker Compose** — оркестрация всех сервисов
- **Nginx** — reverse proxy + SSL (Let's Encrypt / Certbot)

**Docker Compose сервисы:**
```yaml
services:
  api:        # FastAPI REST backend
  bot:        # Telegram bot (python-telegram-bot)
  mongodb:    # MongoDB (векторное хранилище + логи + сессии)
  nginx:      # reverse proxy, SSL termination
```

**Среды:**
- `dev`: локальная машина, Long Polling, MongoDB local
- `prod`: домашний сервер, Webhook, MongoDB с persistent volume

**Требования к вычислениям:**
- CPU: 2+ ядра (embedding на CPU ~1-3 сек/чанк)
- RAM: минимум 4 GB (embedding модель ~1.5 GB + MongoDB ~512 MB)
- Хранилище: 20 GB (модели + MongoDB data + логи)
- GPU: не требуется

**Безопасность:**
- Все секреты (Bot Token, API Keys) — в `.env`, не в репозитории
- `.env` в `.gitignore`
- Whitelist пользователей в MVP (только команда)
- HTTPS обязателен для Webhook
- MongoDB без внешнего порта (только internal Docker network)

## 3. Данные и качество знаний

> ⚠️ Чувствительность данных: контент публичный (открытые сайты, YouTube).
> Отправка в GitHub Models / Gemini API допустима.
> Персональные данные пользователей не собираются.

### 3.1 Сбор и предобработка данных

**Источники и форматы:**

| Источник | Формат | Инструмент |
|---|---|---|
| YouTube | Субтитры JSON/SRT | `youtube-transcript-api` |
| Статьи, блоги | HTML → plain text | `trafilatura` |
| Форумы (Reddit, 4PDA) | HTML → plain text | `trafilatura` |
| Прямой текст | Plain text | без обработки |

**Шаги предобработки:**
1. Извлечение чистого текста
2. Удаление дубликатов абзацев
3. Разбивка на чанки (512 токенов, overlap 50)
4. Добавление метаданных: `{url, source_type, timestamp}`

---

### 3.2 Векторизация и хранение в MongoDB

**Embedding модель:**
- `sentence-transformers/paraphrase-multilingual-mpnet-base-v2`
- Размерность вектора: 768
- Языки: RU + EN + 50 языков
- Размер модели: ~420 MB, работает на CPU

**Схема документа в MongoDB (коллекция `chunks`):**
```json
{
  "_id": "ObjectId",
  "session_id": "uuid",
  "source_url": "https://...",
  "source_type": "youtube | article | forum | text",
  "chunk_index": 3,
  "text": "...текст чанка...",
  "embedding": [0.12, -0.34, ...],  // вектор размерностью 768
  "created_at": "2026-03-01T18:00:00Z"
}
```

**Поиск:** cosine similarity по полю `embedding` (Python-side при малом объёме или
MongoDB Atlas Vector Search при масштабировании).

**Стратегия обновления:**
- Инкрементальное добавление чанков в рамках сессии
- TTL index на поле `created_at` (24 часа) → автоматическое удаление

---

### 3.3 Метрики качества знаний

| Метрика | Цель | Как измеряем |
|---|---|---|
| Покрытие | ≥90% контента из ссылок в базе | Ручная проверка на 5 ссылках |
| Релевантность топ-5 чанков | Содержат ответ на вопрос | Ручная оценка на 20 запросах |
| Дубликаты | < 5% | Скрипт cosine similarity |
| Скорость индексации | < 30 сек на 5 ссылок | Автотест |

## 4. Модель и генерация

### 4.1 Выбор LLM и промптинг

**Основная модель: `GPT-4o mini` через GitHub Models API**
- ✅ Бесплатно по GitHub Student Developer Pack
- ✅ OpenAI-совместимый REST API (легко мигрировать)
- ✅ 128K контекстное окно
- ✅ Отличный русский язык
- ✅ ~2-5 сек на ответ

**Резервная модель: `Gemini 2.0 Flash` (Google AI Studio)**
- ✅ Бесплатно: 1500 req/day, 1M tokens/min
- ✅ Мощнее GPT-4o mini по benchmark'ам 2025
- ✅ Поддержка русского языка
- Используется при исчерпании квоты GitHub Models

**Дешёвая платная альтернатива (если нужно больше):**
- `GPT-4o mini` напрямую через OpenAI: ~$0.15 / 1M input tokens
- При 50 сессиях/день с 5 ссылками — ~$1-3/мес

**Промпт-шаблон:**
```
System:
Ты — эксперт по выбору товаров. Твоя задача — анализировать отзывы,
обзоры и мнения из предоставленных источников и давать структурированную
рекомендацию пользователю.

Правила:
- Основывайся ТОЛЬКО на предоставленном контексте из источников
- Если контекста недостаточно — честно скажи об этом
- Структура ответа строго: Плюсы → Минусы → Итоговая рекомендация
- Отвечай на русском языке
- Не выдумывай факты, которых нет в источниках
- В конце всегда добавляй: "⚠️ Это не экспертный совет."

Контекст из источников:
{retrieved_chunks}

User:
{user_query}
```

**Параметры генерации:**
- `max_tokens`: 1000
- `temperature`: 0.3
- `top_p`: 0.9
- Топ-5 чанков в контексте (~2000 токенов)

---

### 4.2 Контроль качества ответов

**Метрики:**
- Полезность: кнопки 👍/👎 → коллекция `feedback` в MongoDB
- Галлюцинации: ручная проверка 10 запросов/неделю
- Ошибки парсинга: % неуспешных запросов в `interactions.errors`

**Fallback-механизмы:**
| Ситуация | Действие |
|---|---|
| Пустой retrieval | "Не нашёл информации. Добавьте ещё ссылок." |
| Не про товары | "Я помогаю только с выбором товаров." |
| Парсинг провалился | "Не удалось загрузить: {url}" |
| LLM таймаут / ошибка | Retry с Gemini fallback, затем: "Попробуйте ещё раз." |
| GitHub Models лимит | Автоматический switch на Gemini 2.0 Flash |

---

### 4.3 Обучение/дообучение

Fine-tuning не предусмотрен (нет ресурсов и данных в рамках курса).
Используем prompt engineering + RAG как основной механизм качества.

## 5. UX / пользовательский опыт

### 5.1 Сценарии взаимодействия

**Основной сценарий:**
```
Пользователь → "Хочу выбрать беспроводные наушники до 5000 руб"
              + [ссылка YouTube] + [ссылка статья] + [ссылка форум]

Бот → "⏳ Анализирую 3 источника... (15-30 сек)"
Бот → [плюсы / минусы / рекомендация]
      [кнопки: 🔍 Уточнить | 👍 Полезно | 👎 Неточно]
```

**Исключительные сценарии:**
| Ситуация | Ответ бота |
|---|---|
| Ссылка недоступна (403/404) | "Не удалось загрузить {url}. Попробуйте другую." |
| Нет субтитров YouTube | "У видео нет субтитров, пропускаю этот источник." |
| Мало контента | "Источников мало для уверенной рекомендации. Добавьте ещё." |
| Не про товары | "Я специализируюсь на выборе товаров." |
| LLM недоступен | "Сервис временно недоступен, попробуйте через минуту." |

---

### 5.2 Диалоговая логика

**Тип:** Multi-turn в рамках сессии

**Хранение контекста:**
- История последних 5 сообщений хранится в MongoDB (`sessions.history`)
- Между сессиями история не сохраняется (MVP)
- `/new` — начать новую сессию

**Тон:** Дружелюбный, структурированный, без воды.
Эмодзи для навигации (✅ ❌ 🏆 ⚠️). Язык: русский.

---

### 5.3 Метрики UX

| Метрика | Цель | Сбор |
|---|---|---|
| Время ответа | < 30 сек при 5 ссылках | Логи MongoDB `interactions` |
| % полезных ответов | > 70% (👍) | Коллекция `feedback` |
| % ошибок парсинга | < 10% ссылок | `interactions.errors` |
| % сессий с уточнениями | > 30% | Подсчёт по `sessions` |

## 6. Безопасность, соответствие и этика

- **Персональные данные:** Telegram `user_id` хранится только в рамках сессии, не передаётся.
- **Отправка в LLM API:** только публичный контент сайтов/YouTube. Нет личных данных.
- **Прозрачность:** `/start` — бот сообщает: "Я ИИ-помощник. Рекомендации основаны на
  предоставленных источниках и не являются профессиональным советом."
- **Секреты:** Bot Token, API Keys — только в `.env`, в `.gitignore`.
- **MongoDB:** закрытый порт, только internal Docker network, нет внешнего доступа.
