/ основан на шаблоне команды [Reliable ML](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/ML_System_Design_Doc_Template.md) /

## 1. Контекст проекта

### 1.1 Бизнес-задача

**Проблема.** Перед покупкой товара современный потребитель тратит часы на просмотр YouTube-обзоров, чтение тредов на Reddit и сравнительных статей. При этом:
- Большинство видео-обзоров спонсированы и содержат предвзятую информацию.
- Каждый новый источник добавляет лишь 2–5% нового знания, при этом требует 15–30 минут на просмотр.
- Критически важные «нестандартные» мнения (например, единственный честный отзыв с реальным дефектом) теряются в потоке позитивного контента.
- Пользователь вынужден самостоятельно агрегировать и взвешивать противоречивые мнения.

**Решение.** ReviewMind — Telegram-бот (с возможностью перехода в веб-интерфейс), который работает в двух режимах:

**Режим «Авто»** — пользователь называет товар, бот сам ищет и анализирует источники:
1. Автоматически находит и скачивает транскрипты YouTube-видео, посты Reddit и статьи с релевантных сайтов.
2. Индексирует данные в векторной базе для повторного использования — повторный запрос по той же теме возвращает ответ мгновенно.
3. При отсутствии данных в базе — сначала выдаёт быстрый ответ через Tavily (веб-поиск), параллельно запуская полный сбор в фоне.

**Режим «Свои ссылки»** — пользователь сам предоставляет ссылки (YouTube, Reddit, любые статьи), бот парсит именно их и отвечает на основе этого контента.

В обоих режимах RAG-пайплайн приоритетно использует **кураторскую базу знаний** — проверенную командой подборку материалов по популярным категориям товаров, которая обеспечивает качественный ответ даже при скудности других источников.

**Почему без автоматизации не справиться.** Объём видеоконтента на YouTube по популярным товарным категориям исчисляется сотнями часов. Ни один человек не в состоянии систематически и непредвзято агрегировать эту информацию в реальном времени. RAG-подход позволяет сохранять и переиспользовать знания, что принципиально недостижимо при ручном поиске.

**Критерии успеха:**
- Время до первого осмысленного ответа пользователю: < 3 секунды (при наличии кэша) / уведомление о готовности в течение 2–5 минут (при первичном сборе).
- Доля запросов, на которые бот даёт релевантный ответ без перевода на внешний источник: ≥ 80% (после прогрева базы).
- Ответы сопоставимые и/или лучше чем "Обзор от ИИ" при запросе в гугл.

### 1.2 Целевая аудитория и пользователи

**Кто пользуется:** конечные потребители (B2C) — люди, планирующие покупку конкретного товара (электроника, техника, гаджеты, косметика и т.д.) и желающие получить взвешенный анализ без многочасового самостоятельного исследования.

**Пользовательские сценарии:**
- Текстовый чат в Telegram: основной интерфейс MVP.
- Веб-виджет: второй интерфейс, архитектурно поддерживается с первого дня (общий бэкенд).
- Голосовой интерфейс: вне скоупа MVP.

**Предполагаемая нагрузка (MVP):**
- 0–500 активных пользователей в сутки.
- До 5 параллельных сессий сбора данных (фоновые задачи).
- Пиковые часы: вечерние (18:00–23:00).

**Цена ошибки:** средняя. Ошибочный ответ не несёт прямого физического вреда, но подрывает доверие к продукту. Критично: галлюцинации (выдуманные факты о продукте), категоричные утверждения без источника.

### 1.3 Ограничения и допущения

**Что не делаем в MVP:**
- Не интегрируемся с отзывами на Amazon и других локальных маркетплейсах, официальными страницами продуктов (перенесено в Phase 2).
- Не поддерживаем голосовой ввод.
- Не делаем fine-tuning модели — только prompting + RAG.
- Не храним персональные данные пользователей, кроме `telegram_user_id` для идентификации сессии.
- Не строим систему рекомендаций «купить/не купить» — только агрегированный анализ мнений.
- Nginx и webhook-режим — вне скоупа MVP (используется long polling). Добавляются в Phase 2 вместе с веб-интерфейсом.

**Предположения:**
- YouTube Transcript API предоставляет транскрипты для большинства обзорных видео на английском и русском языках.
- Reddit API (PRAW) доступен в рамках free tier для MVP-нагрузки.
- `trafilatura` успешно извлекает текст с большинства открытых веб-страниц без авторизации.
- Контент из открытых источников (YouTube, Reddit, публичные статьи) не содержит персональных данных третьих лиц, требующих специальной обработки по GDPR.
- Данные не являются коммерческой тайной и могут обрабатываться через внешние LLM API.
- На этапе разработки используется **GitHub Student Developer Pack** (бесплатный доступ к GPT-4o mini через GitHub Models API). Переход на production OpenAI API требует только замены `OPENAI_API_KEY` и `OPENAI_BASE_URL` в `.env` — код не меняется.
- Производственный сервер (Intel Xeon E5-2696 v4, amd64) требует подключения к VPN (X-ray VLESS) для доступа к инфраструктуре OpenAI. VPN-подключение обеспечивается на уровне VM до старта контейнеров.

**Репутационные риски:**
- Бот не отвечает на вопросы, не связанные с анализом продуктов (вопросы о политике, медицине, личных темах) — реализуется через системный промпт с явным ограничением домена.
- Все ответы сопровождаются оговоркой об источниках и рекомендацией самостоятельной проверки.
- Спонсорский контент явно помечается флагом `is_sponsored` и понижается в ранжировании — это ключевой этический аспект продукта, декларирующего беспристрастность.

## 2. Архитектура решения

### 2.1 Общая схема

```
┌─────────────────────────────────────────────────────────────────────┐
│                        КЛИЕНТСКИЙ СЛОЙ                              │
│   Telegram Bot (aiogram 3, long polling → webhook в Phase 2)        │
│   Web UI (React — Phase 2)                                          │
└────────────────┬─────────────────────────────────────────────────────┘
                 │  REST / Long polling
┌────────────────▼─────────────────────────────────────────────────────┐
│                     API GATEWAY (FastAPI)                             │
│  POST /query      POST /ingest     GET /status/{job_id}  GET /health │
│  [выбор режима: auto | manual]                                        │
└──────┬──────────────────────────────────────────┬─────────────────────┘
       │                                          │
       ▼                                          ▼
┌──────────────────────┐              ┌───────────────────────────────┐
│   RAG Pipeline        │              │  Job Queue (Redis)            │
│   (Query Layer)       │              │  + Celery Workers             │
│                       │              │  (Ingestion Layer)            │
│ 1. Embed query        │              │                               │
│ 2. Search curated_kb  │              │  [AUTO]  YouTube search       │
│    (score boost ×1.3) │              │          Reddit search        │
│ 3. Search auto_crawled│              │  [BOTH]  trafilatura (URLs)   │
│ 4. Merge + Rerank     │              │          Chunk + Embed        │
│ 5. Tavily fallback    │              │          Upsert to Qdrant     │
│    (if < 3 chunks)    │              │          Notify user          │
│ 6. LLM generate       │              └───────────────────────────────┘
└──────┬───────────────┘
       │
┌──────▼──────────────────────────────────────────────────────────────┐
│                      ХРАНИЛИЩЕ ДАННЫХ                               │
│  Qdrant: [auto_crawled collection] + [curated_kb collection]        │
│  PostgreSQL (метаданные, sources, jobs, query_logs)                  │
│  Redis (кэш сессий 30 мин TTL, Celery broker)                       │
└─────────────────────────────────────────────────────────────────────┘
       │
┌──────▼──────────────────────────────────────────────────────────────┐
│                     ВНЕШНИЕ API                                     │
│  OpenAI: text-embedding-3-small + gpt-4o-mini                       │
│  (dev: GitHub Models API, prod: OpenAI API — только смена ключа)    │
│  YouTube Transcript API  │  YouTube Data API v3                     │
│  Reddit API (PRAW)       │  trafilatura (любые веб-страницы)        │
│  Tavily API (web search fallback, free: 1000 req/мес)               │
└─────────────────────────────────────────────────────────────────────┘
```

**Порядок обработки в режиме «Авто»:**

1. Пользователь выбирает режим `/start` → `[🔍 Авто-поиск]` → задаёт вопрос.
2. FastAPI проверяет оба Qdrant-коллекции: сначала `curated_kb` (приоритет), затем `auto_crawled`.
3. **Достаточно данных (≥ 3 чанка):** немедленный RAG-запрос → ответ < 3 сек, с бейджем `📚` если использованы кураторские источники.
4. **Мало данных (< 3 чанка):** Tavily API возвращает быстрый ответ (~2 сек) из веба → параллельно ставится задача в Celery на полный сбор (YouTube + Reddit). По завершении сбора — обновлённый ответ пушится пользователю.
5. После ответа — пользователь задаёт уточняющие вопросы (multi-turn, история в Redis 30 мин).

**Порядок обработки в режиме «Свои ссылки»:**

1. Пользователь выбирает `[🔗 Свои ссылки]` → присылает URL-адреса (по одному или списком).
2. Бот определяет тип каждого URL: YouTube → `youtube-transcript-api`, Reddit → PRAW, остальное → `trafilatura`.
3. Парсинг выполняется синхронно (для MVP) → чанки → эмбеддинги → Qdrant `auto_crawled` (с `session_id`).
4. Немедленный RAG-запрос по только что загруженным чанкам → ответ.
5. Данные остаются в базе — повторный вопрос по тем же ссылкам работает мгновенно.

### 2.2 Хранилище знаний / документооборот

**Источники знаний (MVP):**

| Источник | Инструмент | Режим | Тип коллекции |
|---|---|---|---|
| YouTube транскрипты | `youtube-transcript-api` | Авто + Ссылки | `auto_crawled` |
| YouTube поиск видео | YouTube Data API v3 | Авто | `auto_crawled` |
| Reddit посты + комментарии | PRAW | Авто + Ссылки | `auto_crawled` |
| Любые веб-страницы | `trafilatura` | Ссылки | `auto_crawled` |
| Веб-поиск (fallback) | Tavily API | Авто (< 3 чанков) | `auto_crawled` (кэш) |
| Кураторская база | Ручное добавление командой | Оба режима | `curated_kb` (постоянная) |

**Предобработка:**
1. Транскрипт/текст очищается от спецсимволов, временных меток, рекламных вставок (детектируются по паттернам: «sponsored by», «используйте промокод»).
2. `trafilatura` автоматически удаляет навигацию, рекламу и HTML с веб-страниц — на выходе только основной текст статьи.
3. Текст сегментируется на чанки по 400–600 токенов с перекрытием 50 токенов (`RecursiveCharacterTextSplitter`).
4. Каждому чанку присваиваются метаданные: `source_type`, `source_id`, `product_query`, `author`, `date`, `language`, `is_sponsored` (bool), `is_curated` (bool), `chunk_index`.

**Кураторская база знаний (`curated_kb`):**
- Постоянная коллекция Qdrant без TTL, наполняется вручную командой через скрипт `scripts/seed_curated_kb.py`.
- Материалы: проверенные статьи с Wirecutter, RTINGS, 4PDA, специализированные гайды по популярным категориям (наушники, смартфоны, ноутбуки, бытовая техника, спортивные товары).
- Требования к источникам: не старше 1 года, авторитетный ресурс, не является рекламным материалом.
- При поиске кураторские чанки получают score boost ×1.3 — они приоритетны при равном семантическом сходстве.
- Если в ответе использованы кураторские источники — бот показывает бейдж `📚 Проверенные источники`.

**Обновление данных:**
- `auto_crawled`: источники сохраняются в PostgreSQL (`source_url`, `parsed_at`). Инкрементальное обновление раз в 30 дней для топ-запросов (Celery Beat). Уже обработанные источники пропускаются по `source_id`.
- `curated_kb`: обновляется вручную по мере появления новых категорий или важных обзоров. Ревью источников не реже раза в месяц.

### 2.3 Интеграции и интерфейсы

| Интеграция | Технология | Назначение |
|---|---|---|
| Telegram Bot | aiogram 3.x, long polling (MVP) / webhook (Phase 2) | Основной UI |
| FastAPI | Python 3.11+, uvicorn | REST API бэкенд |
| OpenAI API | openai SDK | Embeddings + LLM (prod) |
| GitHub Models API | openai SDK + кастомный `base_url` | Embeddings + LLM (dev, бесплатно) |
| YouTube | `youtube-transcript-api`, YouTube Data API v3 | Транскрипты и поиск видео |
| Reddit | PRAW (Python Reddit API Wrapper) | Посты и обсуждения |
| Web scraping | `trafilatura` | Парсинг любых веб-страниц по URL пользователя |
| Web search fallback | Tavily API (free: 1000 req/мес) | Быстрый ответ при пустой базе |
| Qdrant | qdrant-client | Векторный поиск (2 коллекции) |
| PostgreSQL | asyncpg / SQLAlchemy + Alembic | Метаданные, sources, jobs, query_logs |
| Redis | aioredis | Celery broker + кэш сессий (TTL 30 мин) |
| Celery | celery + redis | Фоновые задачи парсинга |
| Celery Flower | flower | Мониторинг задач (внутренняя сеть) |

**Протоколы:**
- Telegram → Bot (long polling): бот сам опрашивает Telegram API (MVP, без домена и Nginx).
- Telegram → FastAPI (webhook, Phase 2): HTTPS POST /webhook/telegram, валидация по `X-Telegram-Bot-Api-Secret-Token`.
- Web UI → FastAPI: REST (HTTPS). Long-polling на `/status/{job_id}` для отслеживания прогресса.
- FastAPI → Celery: Redis message broker.
- Внутренние сервисы: REST по Docker-сети (bot → `http://api:8000`).

**UX в Telegram:**
- `/start` → выбор режима: `[🔍 Авто-поиск] [🔗 Свои ссылки]`.
- **Авто-режим:** пользователь вводит вопрос → бот либо отвечает немедленно (данные есть), либо пишет «⏳ Ищу данные...» и пушит ответ по завершении.
- **Режим ссылок:** бот просит прислать URL → парсит → отвечает.
- Ответ всегда содержит inline-кнопки: `[👍 Полезно] [👎 Не то] [📎 Источники]`.
- Кнопка `📎 Источники` возвращает список URL с пометкой `📚` для кураторских.
- `/mode` — переключение режима в любой момент.

**Мониторинг:**
- Логирование всех запросов/ответов в PostgreSQL (таблица `query_logs`).
- Структурированные логи (structlog → stdout → Grafana/Loki в будущем).
- Celery Flower: мониторинг задач, доступен через SSH-туннель.
- Алерты: ошибки Celery-задач → уведомление в admin Telegram-чат.

### 2.4 Инфраструктура и развёртывание

**Стек:**
- Контейнеризация: **Docker + Docker Compose** (MVP). Каждый сервис — отдельный контейнер.
- `docker-compose.override.yml` для локального dev: открытые порты, hot reload, long polling.
- CI/CD: **GitHub Actions** — автоматическая сборка (buildx, multi-arch), push в GHCR, деплой на сервер по SSH при мерже в `main`.
- Оркестрация: Docker Compose (MVP) → Kubernetes (Phase 3).
- Среды: `dev` (macOS Apple Silicon, arm64) / `prod` (Intel Xeon E5-2696 v4, amd64).

**Совместимость платформ:**
- Все образы собираются с `--platform linux/amd64` для production и `linux/arm64` для локальной разработки.
- Базовые образы (postgres, redis, qdrant) поддерживают multi-arch — совместимость гарантирована.

**Минимальные требования для MVP (prod):**
- VPS/сервер: 2 vCPU, 4 GB RAM, 40 GB SSD.
- GPU: не требуется (embeddings и LLM — через API).
- VPN: на production-сервере необходимо активное подключение X-ray VLESS VPN для доступа к OpenAI API. Проверяется в деплой-скрипте перед стартом контейнеров.

**Безопасность:**
- API-ключи хранятся в `.env` (в `.gitignore`), не в коде.
- В production — Docker secrets или переменные окружения через CI/CD secrets.
- Все сервисы в изолированной Docker-сети; наружу открыт только бот (long polling, MVP) и впоследствии FastAPI через Nginx.
- Rate limiting: FastAPI middleware (slowapi) — не более 10 запросов/мин с одного `user_id`.
- Все данные в изолированных контейнерах PostgreSQL и Qdrant без внешних портов.

**docker-compose.yml (сервисы):**
```yaml
services:
  api:         # FastAPI — REST бэкенд
  bot:         # aiogram bot — long polling (MVP)
  worker:      # Celery worker — парсинг и эмбеддинг
  beat:        # Celery Beat — планировщик обновлений (раз в 30 дней)
  flower:      # Celery Flower — мониторинг задач (internal only)
  qdrant:      # Векторная БД (2 коллекции: auto_crawled, curated_kb)
  postgres:    # Метаданные, логи, sources, jobs
  redis:       # Celery broker + кэш сессий
  # nginx:     # Добавляется в Phase 2 вместе с webhook и веб-интерфейсом
```

## 3. Данные и качество знаний

> **Чувствительность данных:** все источники — публичный контент (YouTube, Reddit). Персональные данные пользователей минимальны (`telegram_user_id`). Контент можно безопасно отправлять через внешние API (OpenAI). При необходимости перехода на закрытые данные — заменить OpenAI на локальную модель (Ollama + nomic-embed-text).

### 3.1 Сбор и предобработка данных

**Источники и форматы:**

| Источник | Формат | Инструмент | Режим запуска |
|---|---|---|---|
| YouTube транскрипты | JSON/SRT → plain text | `youtube-transcript-api` | Авто + Ссылки |
| YouTube поиск | JSON (Data API) | YouTube Data API v3 | Только Авто |
| Reddit посты + комментарии | JSON → plain text | PRAW | Авто + Ссылки |
| Веб-страницы по URL | HTML → plain text | `trafilatura` | Только Ссылки |
| Веб-поиск (fallback) | JSON (LLM-ready text) | Tavily API | Авто (< 3 чанков) |
| Кураторские материалы | Markdown / plain text | Ручное добавление | Инициализация + обновление |

**Шаги предобработки:**
1. **Извлечение текста:** `trafilatura` удаляет навигацию, рекламу, HTML; YouTube-транскрипты очищаются от временны́х меток и повторяющихся фраз; Reddit-комментарии фильтруются по score (топ по полезности).
2. **Фильтрация спонсорства:** regex-паттерны → `is_sponsored=True` если > 15% текста содержит коммерческие триггеры. Контент не удаляется, но помечается и понижается в ранжировании.
3. **Фильтрация качества:** чанки < 50 токенов — drop; транскрипты < 500 слов — skip.
4. **Дедупликация:** cosine similarity > 0.95 между новым и существующим чанком → skip upsert.
5. **Сегментация:** `RecursiveCharacterTextSplitter`, 400–600 токенов, overlap 50 токенов.
6. **Метаданные:** `{product_query, source_type, source_id, url, author, published_date, language, is_sponsored, is_curated, chunk_index, session_id}`.

### 3.2 Векторизация и индексирование

| Параметр | Значение |
|---|---|
| Модель embedding | `text-embedding-3-small` (OpenAI / GitHub Models) |
| Размерность | 1536 |
| Метрика | Cosine similarity |
| Хранилище | Qdrant (self-hosted) |
| Коллекция 1 | `auto_crawled` — контент, собранный ботом и пользователем |
| Коллекция 2 | `curated_kb` — кураторская база команды, постоянная, без TTL |
| Score boost | `curated_kb` × 1.3 при объединении результатов |
| Payload filters | `product_query`, `source_type`, `language`, `is_sponsored`, `session_id` |
| Обновление индекса | Инкрементальный upsert по хэшу текста + `source_id` + `chunk_index` |

**Стратегия поиска (гибридный retrieval):**
1. Параллельный поиск в обеих коллекциях — Top-K=5 из каждой.
2. Объединение результатов: чанки из `curated_kb` получают score ×1.3.
3. Reranking: `is_sponsored=True` → score ×0.7.
4. Итоговый контекст: топ-8 чанков после ранжирования.

**Стратегия обновления:**
- `auto_crawled`: инкрементальный upsert, старые версии чанков заменяются при повторном парсинге.
- `curated_kb`: добавление через скрипт `seed_curated_kb.py`, пересборка только при смене модели эмбеддинга.
- При смене embedding-модели: создаётся новая версия коллекции, миграция выполняется поэтапно.

### 3.3 Метрики качества знаний

| Метрика | Цель MVP | Как измеряется |
|---|---|---|
| Покрытие тем (`auto_crawled`) | ≥ 10 источников на запрос | Автоматически при парсинге |
| Покрытие кураторской базы | ≥ 5 категорий товаров при запуске | Подсчёт категорий в `curated_kb` |
| Актуальность `auto_crawled` | Обновление ≤ 30 дней | PostgreSQL `parsed_at` |
| Актуальность `curated_kb` | Ревью ≤ 30 дней | `updated_at` в payload |
| Отсутствие дублей | < 5% дублирующих чанков | Cosine sim > 0.95 при upsert |
| Точность поиска | Top-5 чанков релевантны запросу | Ручная проверка выборки (20 запросов) |

## 4. Модель и генерация

### 4.1 Выбор LLM и промптинг

**Модель:** `gpt-4o-mini`

**Стратегия API:**
- **Разработка:** GitHub Models API (бесплатно по GitHub Student Developer Pack). Тот же OpenAI-совместимый REST, отличается только `base_url` и ключ. Позволяет разрабатывать и тестировать с нулевыми затратами.
- **Production:** OpenAI API напрямую (`api.openai.com`). Переключение — одна строка в `.env`, код не меняется.
- **Резерв при экономии:** `deepseek-chat` через OpenRouter ($0.07/1M токенов) — подключается той же заменой `base_url`.

**Обоснование выбора gpt-4o-mini:**
- Стоимость: $0.15/1M input tokens, $0.60/1M output tokens.
- Контекстное окно: 128K токенов — достаточно для 8–10 чанков контекста.
- Качество следования сложным структурированным промптам: достаточное для задачи агрегации.
- `o4-mini` (reasoning) избыточен — reasoning-модели решают многошаговые логические задачи, а не синтез текста.

**Системный промпт (шаблон):**
```
Ты — беспристрастный аналитик потребительских отзывов. Твоя задача — агрегировать
мнения из реальных источников и предоставлять сбалансированный анализ.

ПРАВИЛА:
1. Отвечай ТОЛЬКО на основе предоставленного контекста. Не выдумывай факты.
2. Явно выделяй противоречивые мнения. Если есть критические отзывы — упомяни их.
3. Источники с флагом [sponsored] считай менее достоверными, указывай это в анализе.
4. Структурируй ответ: [✅ Плюсы] [❌ Минусы] [⚖️ Спорные моменты] [🏆 Вывод].
5. Если контекста недостаточно — честно скажи об этом, не додумывай.
6. Отвечай на языке пользователя.
7. В конце укажи: количество источников и наличие проверенных материалов (📚).

КОНТЕКСТ:
{retrieved_chunks}

ИСТОРИЯ ДИАЛОГА:
{chat_history}

ВОПРОС ПОЛЬЗОВАТЕЛЯ:
{user_query}
```

**Параметры генерации:** temperature=0.3, max_tokens=1000, top_p=0.9.

**RAG-стратегия:**
- Retrieval: параллельный поиск в `curated_kb` (Top-5) и `auto_crawled` (Top-5).
- Merge + rerank: кураторские чанки ×1.3, спонсорские ×0.7, итого Top-8.
- Tavily fallback: если после rerank < 3 чанков с confidence > 0.75 → синхронный вызов Tavily API, результаты добавляются в контекст и уписываются в `auto_crawled`.
- Pre-filter: по `product_query` payload — только чанки по нужному продукту.

### 4.2 Контроль качества ответов

**Метрики:**
- Пользовательский рейтинг (👍/👎) — собирается после каждого ответа, хранится в PostgreSQL.
- Доля ответов «недостаточно данных»: целевой показатель < 10% после прогрева кураторской базы.
- Доля галлюцинаций: ручной ревью выборки (5% запросов еженедельно).
- % запросов, закрытых Tavily без Celery-задачи: отслеживается для оценки полноты кураторской базы.

**Механизмы защиты от ошибок:**

| Ситуация | Действие |
|---|---|
| `curated_kb` + `auto_crawled` → ≥ 3 чанка | Немедленный RAG-ответ |
| Суммарно < 3 чанков (авто-режим) | Tavily fallback → быстрый ответ + фоновый Celery |
| Суммарно < 3 чанков (режим ссылок) | «Недостаточно данных в ваших источниках. Добавьте ещё ссылок.» |
| Запрос не по теме товаров | Guard-prompt → «Я специализируюсь только на анализе товаров» |
| trafilatura не смогла спарсить URL | Уведомление пользователя, skip, остальные ссылки обрабатываются |
| LLM API недоступен | Retry ×3 с экспоненциальным backoff → «Сервис временно недоступен» |
| Celery-задача упала | Retry ×3 → алерт в admin-чат → ответ пользователю с извинением |

Все ответы сопровождаются кнопкой `📎 Источники` — полный список URL с пометкой `📚` для кураторских материалов.

### 4.3 Обучение/дообучение

В MVP fine-tuning не предусмотрен. Только промпт-инжиниринг и RAG.

Версионирование: промпты хранятся в репозитории (Git), откатываются вместе с кодом. При смене модели — тестирование на наборе из 50 эталонных Q&A пар вручную.

## 5. UX / пользовательский опыт

### 5.1 Сценарии взаимодействия

**Основные сценарии:**

| Сценарий | Поведение бота |
|---|---|
| `/start` | Приветствие + выбор режима: `[🔍 Авто-поиск] [🔗 Свои ссылки]` |
| Авто: данные есть в базе | Немедленный ответ < 3 сек, `📚` если кураторские |
| Авто: кураторская база покрывает тему | Быстрый ответ без парсинга, с бейджем `📚 Проверенные источники` |
| Авто: нет данных → Tavily | Быстрый ответ (~2 сек) из веб-поиска + параллельный полный сбор в фоне |
| Авто: нет данных → Celery | «⏳ Ищу данные (~3 мин)...» → push-уведомление по завершении |
| Ссылки: пользователь прислал URL | Парсинг (YT/Reddit/Web) → ответ по предоставленным источникам |
| Уточняющий вопрос | Multi-turn с контекстом предыдущего ответа (история в Redis) |
| Сравнение товаров | «iPhone 16 vs Samsung S25» → структурированная таблица сравнения |
| `/mode` | Переключение между авто и ручным режимом |

**Исключительные сценарии:**

| Ситуация | Поведение |
|---|---|
| Тема не по продуктам | «Я помогаю только с анализом товаров. Попробуйте спросить о конкретном продукте.» |
| URL недоступен / paywall | «Не удалось загрузить: {url}. Попробуйте другую ссылку.» |
| YouTube без субтитров | «У видео нет субтитров, пропускаю этот источник.» |
| Ошибка LLM API | Retry ×3 → «Сервис временно недоступен. Попробуйте через несколько минут.» |
| Непонятный запрос | «Уточните, пожалуйста: какой именно продукт вас интересует?» |

### 5.2 Диалоговая логика

- **Режим пользователя:** выбранный режим (авто/ссылки) сохраняется в Redis-сессии — не нужно переключаться при каждом сообщении. Команда `/mode` переключает режим.
- **Single-turn:** каждый вопрос — независимый RAG-запрос с полным контекстом.
- **Multi-turn:** история последних 5 сообщений хранится в Redis с TTL 30 минут. Передаётся в LLM как `chat_history`. В режиме ссылок — в контексте остаются чанки из всех ссылок текущей сессии.
- **История пользователя:** сохраняется в PostgreSQL для аналитики (только `telegram_user_id` + запрос + оценка, без идентификации личности).

**Тон ответов:** нейтральный, фактический. Эмодзи только в структурных маркерах (✅ ❌ ⚖️ 🏆 📚). Язык ответа определяется автоматически по языку запроса пользователя (`langdetect`).

### 5.3 Метрики UX

| Метрика | Цель |
|---|---|
| Время ответа (данные есть) | < 3 сек |
| Время ответа через Tavily fallback | < 5 сек |
| Время уведомления (Celery-сбор) | < 5 мин |
| % запросов закрытых без Celery (кэш + Tavily) | ≥ 60% (после прогрева базы) |
| % сессий с уточняющими вопросами | ≥ 30% (показатель вовлечённости) |

Логирование: каждый запрос, ответ, режим, оценка и source-list пишутся в `query_logs`.

## 6. Безопасность, соответствие и этика

- **Персональные данные:** хранится только `telegram_user_id` (не имя, не телефон). Данные используются исключительно для идентификации сессии и аналитики. Политика конфиденциальности публикуется в боте.
- **Прозрачность:** при первом запуске бот явно сообщает, что является AI-системой и что его ответы основаны на публичных источниках.
- **Фильтрация:** запросы, не связанные с анализом товаров, отклоняются на уровне системного промпта.
- **Авторские права:** система использует только транскрипты и текстовые фрагменты (не копирует полный контент), предоставляет ссылки на источники. Fair use / information aggregation.
- **Спонсорский контент:** явно помечается в ответах — этически важный аспект для продукта, декларирующего беспристрастность.
- **Rate limiting:** защита от злоупотреблений (10 запросов/мин на пользователя).
- **GDPR:** минимальный сбор данных, возможность удаления истории по запросу (`/delete_my_data`).

## 7. План внедрения и эксплуатации

### 7.1 Этапы проекта

Проект строится последовательно: каждый этап имеет чёткие критерии готовности, и переход к следующему происходит только после верификации текущего. Это позволяет отлаживать систему пошагово, избегая ситуации «всё не работает и непонятно где».

| Этап | Описание | Ключевой результат |
|---|---|---|
| **0. Фундамент и CI/CD** | Репозиторий, Docker Compose, GitHub Actions, проверка совместимости arm64/amd64 | `git push` → автодеплой, все контейнеры зелёные |
| **1. Telegram ↔ LLM чат** | aiogram (long polling) + FastAPI + OpenAI клиент (GitHub Models) | Вопрос в Telegram → ответ LLM < 5 сек |
| **2. Слой данных** | PostgreSQL (миграции), Qdrant (2 коллекции: `auto_crawled` + `curated_kb`), Redis | Все БД healthy, smoke-тесты проходят |
| **3. Инструменты парсинга** | YouTube (`youtube-transcript-api`), Reddit (PRAW), Web (`trafilatura`) как изолированные модули | Каждый парсер работает автономно, unit-тесты проходят |
| **4. RAG-пайплайн** | Embed → Qdrant upsert → гибридный поиск → LLM генерация, seed-данные для теста | Запрос по seed-продукту возвращает релевантный ответ |
| **5. Ручной режим (ссылки)** | Бот принимает URL → парсит → отвечает по предоставленным источникам | Пользователь присылает 3 ссылки → получает анализ |
| **6. Celery async** | Фоновые задачи, уведомления по завершении, Celery Flower | Парсинг 5 ссылок → немедленное «жду...» → push-ответ через ~3 мин |
| **7. Доп. фичи** | Кураторская база (`curated_kb` seed), Tavily fallback | Популярная категория → ответ с `📚`; нишевый товар → ответ через Tavily |
| **8. Авто-режим** | Извлечение продукта → автопоиск YouTube + Reddit + полный pipeline | Только название товара → полный анализ без ссылок от пользователя |

**Чек-лист готовности к запуску MVP (этапы 0–6):**
- [ ] GitHub Actions: lint + тесты на PR, автодеплой на merge в main
- [ ] Контейнеры собираются и работают на arm64 (dev) и amd64 (prod)
- [ ] VPN-проверка в деплой-скрипте (доступность OpenAI API)
- [ ] Telegram-бот: long polling, /start с выбором режима
- [ ] PostgreSQL: все миграции применены, таблицы созданы
- [ ] Qdrant: коллекции `auto_crawled` и `curated_kb` инициализированы
- [ ] Парсеры YouTube / Reddit / trafilatura: unit-тесты зелёные
- [ ] RAG: ответы цитируют реальные источники, не галлюцинируют
- [ ] Режим ссылок: принимает URL всех трёх типов
- [ ] Celery + Flower: задачи выполняются, ошибки алертятся в admin-чат
- [ ] Rate limiting: 10 запросов/мин на `user_id`

### 7.2 Поддержка и эксплуатация

- **Мониторинг:** structlog → stdout; Celery Flower (SSH-туннель для доступа); алерты через Telegram.
- **Обновление `auto_crawled`:** Celery Beat раз в 30 дней пересчитывает топ-50 запросов.
- **Обновление `curated_kb`:** ручное добавление командой через `scripts/seed_curated_kb.py`, ревью раз в месяц.
- **Деплой:** `git push main` → GitHub Actions → `docker compose pull && docker compose up -d` на сервере.
- **Метрики эксплуатации:** latency P50/P95, cost per session (API costs), Celery task success rate, доля Tavily-ответов.

## 10. Приложения

### Словарь терминов
- **RAG (Retrieval-Augmented Generation)** — подход, при котором LLM получает релевантные фрагменты из базы знаний вместо того, чтобы полагаться только на параметрические знания.
- **Qdrant** — open-source векторная база данных, оптимизированная для ANN-поиска по embedding-векторам.
- **Embedding** — числовое векторное представление текста, сохраняющее семантическое сходство.
- **Celery** — Python-фреймворк для распределённой асинхронной обработки задач.
- **Чанк (chunk)** — фрагмент текста, получаемый при сегментации документа для векторизации.
- **`auto_crawled`** — Qdrant-коллекция с контентом, собранным ботом автоматически или предоставленным пользователем.
- **`curated_kb`** — Qdrant-коллекция с кураторскими материалами команды, постоянная, приоритетная при поиске.
- **Score boost** — искусственное повышение релевантности кураторских чанков (×1.3) при объединении результатов поиска.
- **Tavily** — API для веб-поиска, оптимизированный для LLM-пайплайнов: возвращает чистый текст, не сырой HTML.
- **trafilatura** — Python-библиотека для извлечения основного текста с веб-страниц (удаляет рекламу, навигацию, HTML).
- **aiogram** — async Python-библиотека для Telegram Bot API.
- **PRAW** — Python Reddit API Wrapper.
- **Long polling** — режим работы Telegram-бота: бот сам опрашивает серверы Telegram. Не требует домена и HTTPS.
- **GitHub Models API** — бесплатный доступ к GPT-4o mini через GitHub Student Developer Pack (OpenAI-совместимый REST).
